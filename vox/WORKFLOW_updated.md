## Session Start — Pipeline Briefing

If you paste a **Vox Pipeline Briefing** at the start of a session, do this first:

1. Acknowledge it — confirm which roles are active, which are flagged for follow-up, and what the most recent eval was.
2. Use it as the live source of truth for the entire session. It overrides pipeline state stored in memory.
3. Surface any stale follow-up alerts or rejection patterns proactively before moving to the task at hand.
4. Do not ask the user to re-explain the pipeline — the briefing is the briefing.

The briefing is generated by **Veritas** (your role evaluation and pipeline tracking tool). If no briefing is pasted, use the pipeline state in the context files as the baseline.

## Cover Letter Workflow — ALWAYS follow this order

1. **Build** — Write a Node.js script using the `docx` npm package. Use the established template (see STYLE_GUIDE.md). Save output to `/mnt/user-data/outputs/`.
2. **Validate** — Run `python /mnt/skills/public/docx/scripts/office/validate.py` on the output file.
3. **Preview** — Call `present_files` to show the .docx in the side panel. Stop here and wait for feedback. Adjust this path if not using the Claude computer use environment.
4. **Iterate** — Make requested edits using `str_replace` on the JS script, regenerate, re-validate, re-present.
5. **Drive** — Only after Mike approves, upload to Google Drive using `tool_search` to load Drive tools, then `Google Drive:create_file`.

**Never skip to Drive first. The side panel preview is the review step.**

## Resume Workflow

- The base resume does not persist between sessions as a local file. Always fetch the current version from Google Drive before editing.
- Search Drive for `[YOUR_NAME]_Resume.docx` to get the base. Download, edit, validate, present, then re-upload.
- Output naming: `[YOUR_NAME]_Resume.docx` (base) or `[YOUR_NAME]_[Company]_Resume.docx` (role-specific variant).
- Same build → preview → Drive order applies.

## When Starting a New Cover Letter

1. Read the job description carefully.
2. Search past conversations for any prior research on the company.
3. Check project knowledge for resume content and writing rules.
4. If a pipeline briefing was pasted at session start, check whether this company appears in it — eval score, gaps, and recommendation inform the letter's framing.
5. Build the .docx — do not draft in plain text first.
